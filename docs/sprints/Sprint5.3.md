# Storage format (important)

This is where I recommend a small architectural improvement.

Instead of storing only vectors, store the metadata together with each vector, for example:

```json
[
  {
    "chunkId": "DOC-7E311A25-C001",
    "embedding": [0.021, -0.443, ...]
  },
  {
    "chunkId": "DOC-7E311A25-C002",
    "embedding": [...]
  }
]
```

Later, when we build the vector index, we can simply read this file and know exactly which embedding belongs to which chunk without joining two different files.

# 4. Architectural Rule

I would add this to the project documentation.

## Document Identity Rule

> The original engineering evidence keeps its human-readable filename inside `knowledge_base/`.
>
> Once registered, every downstream artifact must be identified by the immutable `documentId`.
>
> This includes:
>
> * chunk files
> * embedding files
> * extracted knowledge
> * graph import files
> * Neo4j nodes
> * traceability records
>
> The original filename is stored only as metadata for auditing and user display.

---

## Why this matters

This avoids collisions such as:

```
Graph.pdf
Graph.pdf
Graph.pdf
```

uploaded by different engineers or at different times.

Each becomes:

```
DOC-7E311A25
DOC-AB19F402
DOC-C923117A
```

while still preserving the original filename in the metadata.
