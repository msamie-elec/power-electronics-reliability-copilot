# Version 0.5.0 — AI Knowledge Engineering Platform

## Overview

Version 0.5.0 marks the transition from a manually created Knowledge Graph to an automated, evidence-driven AI knowledge engineering platform.

Previous releases established the core foundations:

- v0.1 — User Interface
- v0.2 — Backend Services
- v0.3 — Retrieval-Augmented Generation (RAG)
- v0.4 — Knowledge Graph Foundation

Version 0.5 builds on these foundations by introducing an automated document processing pipeline capable of transforming engineering documents into structured knowledge suitable for Retrieval-Augmented Generation (RAG), Knowledge Graphs, and GraphRAG.

Rather than relying on manually curated graph data, the system will automatically extract engineering knowledge from uploaded technical documents, populate the Neo4j knowledge graph, and combine semantic retrieval with graph reasoning to provide evidence-backed engineering recommendations.

---

# Version Objective

The primary objective of v0.5 is to implement an end-to-end AI knowledge engineering pipeline capable of:

- Processing engineering documents
- Creating semantic document representations
- Extracting structured engineering knowledge
- Automatically expanding the Neo4j Knowledge Graph
- Combining vector search and graph traversal
- Producing explainable, evidence-backed AI answers

By the end of this version, the system will have evolved from a Retrieval-Augmented Generation prototype into a complete GraphRAG platform suitable for enterprise engineering applications.

---

# Architectural Evolution

Previous versions followed a traditional RAG workflow:

Engineering PDF
↓

Chunking
↓

Embeddings
↓

Vector Search
↓

LLM Answer

Version 0.5 introduces an evidence-centred GraphRAG architecture:

Engineering PDF

↓

LlamaIndex

↓

Chunking

↓

Embeddings

↓

FAISS

↓

Retrieve Relevant Chunks

↓

Extract Engineering Entities

↓

Neo4j Traversal

↓

Hybrid Retrieval

↓

GPT Reasoning

↓

Evidence-backed Engineering Answer

This architecture allows engineering knowledge to be continuously extracted from real documents while maintaining traceable evidence throughout the reasoning process.

---

# Dual Document Ingestion Strategy

Version 0.5 introduces two independent document ingestion workflows.

## 1. User Document Pipeline

Purpose:

Process engineering documents uploaded by users during investigations.

Examples:

- Maintenance reports
- Failure reports
- Test reports
- Inspection documents
- Customer documentation

Primary objective:

Provide fast semantic retrieval for engineering investigations.

Pipeline:

Upload

↓

Document Registration

↓

Chunking

↓

Embeddings

↓

FAISS

↓

Retrieval

↓

LLM Answer

Future enhancement:

This retrieval pipeline will also incorporate BM25 keyword search to complement semantic retrieval, enabling accurate matching of engineering terminology, component identifiers, numerical values and technical specifications.

---

## 2. Evidence Knowledge Pipeline

Purpose:

Build and continuously expand the engineering knowledge base.

Examples:

- Technical standards
- Reliability handbooks
- Manufacturer manuals
- Scientific publications
- Engineering textbooks
- Internal engineering knowledge

Pipeline:

Upload

↓

Document Registration

↓

Chunking

↓

Embeddings

↓

Knowledge Extraction

↓

Graph-ready JSON

↓

Neo4j Population

↓

GraphRAG

↓

Enterprise Knowledge Base

Only trusted documents approved by administrators will enter this pipeline.

### Frozen design decisions

* ✅ The existing **Power Electronics Reliability Copilot** remains the project's front-end identity.
* ✅ We consider the **Copilot** to have been established in **v0.3**.
* ✅ v0.4 provides the **Knowledge Graph Foundation**.

* ✅ We postpone renaming Version 0.5 and related documentation until the documentation revision phase.

I also think this makes the project tell a much stronger engineering story. Someone looking at the GitHub repository or your portfolio will see a clear progression:

**UI → Backend → RAG Copilot → Knowledge Graph → Evidence-backed Reasoning → Agentic AI (future)**

That's a coherent architecture evolution that reflects how real enterprise AI systems mature.

From this point onward, we'll implement against this frozen roadmap unless there's a compelling technical reason to adjust it.
